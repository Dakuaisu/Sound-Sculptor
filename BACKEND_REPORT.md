# Sound Sculptor — Backend Analysis Report
*Read-only audit. No code was modified. Findings adversarially verified to remove false positives.*

**Totals:** 2 P0 (critical) · 22 P1 (important) · 42 P2 (nice-to-have)

## Executive Summary

The Sound Sculptor Flask backend is a small, readable app-factory codebase (three blueprints under `/api`, a per-request spotipy client, a lazy KNN model) that works on the happy path but is **not production-safe today**. The single most urgent problem is a live credential leak: `server/.cache` is **tracked in git** (verified via `git ls-files` and `git show HEAD:server/.cache`) and contains a real Spotify `access_token` and a long-lived `refresh_token` with full granted scope. Anyone with repo access can mint access tokens against the victim's account until the grant is manually revoked at Spotify. This is a "rotate now, scrub history" incident, not a code fix you can ship calmly. It is compounded by `SECRET_KEY` defaulting to the public literal `dev-fallback-change-me` (`config.py:9`) and by the **entire** token_info (access + refresh) being stored in a signed-but-unencrypted client cookie (`auth.py:30`) — so if the prod key is unset, session cookies are both readable and forgeable, i.e. a full auth bypass.

Beyond the secrets, the dominant theme is **missing error handling around every external call**. There is no `@app.errorhandler` anywhere (`app.py`), and only `/predict` wraps its work in try/except. The headline AI feature (`ai.py::generate`) calls OpenAI, then runs 10-30 **serial** `sp.search()` round-trips, then creates and populates a **public** playlist — all unguarded. Any OpenAI rate-limit or Spotify 5xx mid-flight returns a raw 500, and a failure after `user_playlist_create` orphans an empty public playlist in the user's account. The same unguarded pattern affects the live `/callback`, `/me`, `/create-playlist`, and the token-refresh path in `get_token()` (a revoked/expired refresh token yields a 500 instead of a clean 401 → reconnect).

The remaining themes are **reliability/perf** (nginx `proxy_read_timeout 60s` < gunicorn `--timeout 120s` so slow generations 504 while the worker keeps running; serial search loop saturates one of only two sync workers; full `tracks_features.csv` read just to keep the `id` column; no rate limiting on the OpenAI-cost endpoint), a **model-correctness bug** (KNN runs on raw unscaled features so tempo ~40-220 and loudness ~-60-0 dominate the five 0-1 sliders — the manual feature is effectively broken), and a long tail of **maintainability/dead-code** issues (dead `/user-data` and `/save-discover-weekly` endpoints, duplicated get-client-or-401 boilerplate, commented-out traps in `config.py`/`spotify.py`, no tests, no logging config, manifest/version drift). Note the slider flow also returns 503 out of the box because `model.pkl`/`tracks_features.csv` are gitignored and absent.

# 1. Architecture

## ARCHITECTURE — Sound Sculptor Flask Backend

Sound Sculptor is an AI/ML Spotify playlist generator. The backend is a Flask application factory exposing a JSON `/api/*` surface, fronted by nginx and run under gunicorn in production. It integrates three external systems: **Spotify** (OAuth + Web API via `spotipy`), **OpenAI** (`gpt-3.5-turbo` for prompt-to-songs), and a local **scikit-learn KNN** model for audio-feature recommendations.

### (a) Folder structure

```
Sound-Sculptor/                     (repo root = working dir)
├── server/                         Python/Flask backend package
│   ├── app.py                      create_app() factory, CORS, ProxyFix, blueprint reg, /api/health
│   ├── config.py                   Config class, dotenv load, secrets, scopes, CORS origins
│   ├── run.py                      dev entry: create_app() + app.run(debug=True, port=5000)
│   ├── .cache                      *** TRACKED IN GIT — leaked Spotify access+refresh token ***
│   ├── blueprints/
│   │   ├── auth.py                 /api/connect, /api/callback, /api/me
│   │   ├── playlist.py             /api/predict, /api/save-discover-weekly, /api/user-data, /api/create-playlist
│   │   └── ai.py                   /api/ai/generate, /api/ai/save
│   ├── services/
│   │   ├── spotify.py              SpotifyOAuth builder, get_token() refresh, get_spotify_client()
│   │   └── ml.py                   lazy KNN load, predict_songs()
```

### (b) Module responsibilities

- **`server/app.py` — `create_app(config=None)`** (lines 7-34): the application factory. Loads `Config` (`from_object`), wraps WSGI in `ProxyFix(x_for=1, x_proto=1, x_host=1)` so `url_for(..., _external=True)` honors nginx `X-Forwarded-*` headers, configures flask-cors with `supports_credentials=True` over `CORS_ORIGINS` (methods GET/POST/OPTIONS), registers the three blueprints, and defines the only inline route `/api/health` → `{"status":"ok"}`.

- **`server/config.py` — `Config`** (lines 8-31): `load_dotenv()` at import, then a class-attribute config. `SECRET_KEY` falls back to literal `'dev-fallback-change-me'` (line 9 — forgeable sessions if unset in prod). `SESSION_COOKIE_NAME='sound_sculptor_cookie'`. Spotify creds read from env `CLIENT_ID`/`CLIENT_SECRET`. `SPOTIFY_SCOPES` (lines 14-19): `user-library-read playlist-read-private user-top-read playlist-modify-public` (only `-modify-public`, so created playlists are public; `user-top-read`/`user-library-read` are requested but unused by any endpoint). `OPENAI_API_KEY`, `CORS_ORIGINS` (comma-split), `FRONTEND_URL` (default `localhost:5173`). Lines 28-31 are dead commented `current_app` debug prints (would have crashed at class-body eval — no app context).

- **`server/run.py`** (lines 1-22): dev-only entrypoint. Inserts repo root on `sys.path`, builds `app = create_app()`, and under `__main__` runs `app.run(debug=True, port=5000)`. **Dev only** — production never executes this (gunicorn imports `server.app:create_app()` directly).

- **`server/services/spotify.py`**: the Spotify integration layer.
  - `get_spotify_oauth()` (lines 8-18): builds `SpotifyOAuth` from config with `redirect_uri=url_for('auth.callback', _external=True)` and explicitly `cache_path=None, cache_handler=None`. A commented-out duplicate of this function sits at lines 20-28. **Critical:** passing `cache_path=None` does **not** disable caching — spotipy falls back to a default `CacheFileHandler` that writes `.cache` in the CWD, which is exactly how the leaked `server/.cache` token file was produced and committed.
  - `get_token()` (lines 31-47): reads `token_info` from the Flask `session`; raises `PermissionError` if absent; if `expires_at - now < 60s`, calls `oauth.refresh_access_token(refresh_token)` and writes the refreshed dict back to `session`.
  - `get_spotify_client()` (lines 50-53): returns `spotipy.Spotify(auth=token_info['access_token'])` — a **fresh client per request**, no global client.

- **`server/services/ml.py`**: the KNN recommender.
  - Module globals `_model`, `_y_train`; `FEATURE_KEYS` = `[danceability, energy, loudness, acousticness, instrumentalness, tempo, liveness]` (7 features, lines 13-21).
  - `_load_model()` (lines 28-49): lazy singleton load; `joblib.load(model.pkl)` and `pd.read_csv(tracks_features.csv)` then keeps only `_y_train = train_data['id']` — it reads the **entire** CSV just to recover the id column.
  - `predict_songs(features)` (lines 52-74): validates keys, builds a `(1,7)` array in `FEATURE_KEYS` order, calls `_model.kneighbors(input_arr)`, returns `_y_train.iloc[indices[0]]` track IDs. **Correctness concern:** `kneighbors` runs on **raw, unscaled** features — `tempo` (~40-220) and `loudness` (~-60-0) dominate the Euclidean distance over the 0-1 features (danceability/energy/etc.); no `StandardScaler`.

- **Blueprints** (all `url_prefix='/api'`, except `ai_bp` = `/api/ai`):
  - **`auth.py`** — `connect()` clears session and 302-redirects to Spotify's authorize URL; `callback()` clears session, exchanges `code` via `get_access_token(check_cache=False)`, stores full `token_info` in session, redirects to `{FRONTEND_URL}/choice`; `me()` returns id/display_name/images (401 on `PermissionError`).
  - **`playlist.py`** — `predict()` (ML, no auth), `save_discover_weekly()`, `user-data`, `create_playlist()`. The Spotify-backed routes guard `get_spotify_client()` with `try/except PermissionError → 401` but the actual Spotify calls (`current_user`, `user_playlist_create`, `user_playlist_add_tracks`) are unguarded.
  - **`ai.py`** — `_parse_songs_from_text()` (JSON-first then 3 regex patterns), `generate()` (OpenAI → parse → serial Spotify search → create public playlist; stashes `session['playlist_id']`), `save()` (`current_user_follow_playlist`). **`generate()` has NO try/except around the OpenAI call or any Spotify call**, and does a **serial `sp.search()` per song** (10-30 sequential round-trips).

### (c) Request flow (production)

```

   ├─ flask-cors (CORS_ORIGINS, supports_credentials)
   ▼
Blueprint route (auth / playlist / ai)
   ▼
Service layer (services/spotify.py → spotipy → Spotify Web API ; services/ml.py → KNN ; ai.py → OpenAI)
```

Both gunicorn and nginx run in the **same container** (`docker-entrypoint.sh`: gunicorn backgrounded, nginx foreground). The dev compose profile instead runs `python -m server.run` (Flask debug:5000) + Vite (:5173).

### (d) Authentication flow (Spotify OAuth, session-cookie based)

```
1. Browser → GET /api/connect   (frontend Connect.jsx uses a plain <a href="/api/connect">,
                                  a full-page navigation — NOT the fetch api.js client)
        auth.connect(): session.clear(); 302 → Spotify authorize URL
2. User authorizes on accounts.spotify.com (scopes from Config.SPOTIFY_SCOPES)
3. Spotify → 302 → GET /api/callback?code=...   (redirect_uri = url_for('auth.callback', _external=True))
        auth.callback(): session.clear();
                         token_info = oauth.get_access_token(code, check_cache=False)
                         session[TOKEN_INFO='token_info'] = token_info   ← full dict (access+refresh)
                         302 → {FRONTEND_URL}/choice
4. Subsequent /api/* calls send the signed 'sound_sculptor_cookie'
        get_token(): if expiring within 60s → oauth.refresh_access_token(refresh_token),
                     rewrite session
        get_spotify_client(): spotipy.Spotify(auth=access_token)
```

The **entire `token_info` (access + refresh tokens) lives in the Flask session cookie**, which is *signed but NOT encrypted* — its base64 payload is readable by anyone holding the cookie. Combined with the `SECRET_KEY` default fallback, an unset prod key makes the cookie forgeable. Auth state is fully client-cookie-driven; there is no server-side session store. There is **no logout endpoint** (only `session.clear()` at the start of `connect`/`callback`).

### (e) Spotify integration flow (per-request client)

Every Spotify-touching request builds a brand-new client: `get_spotify_client()` → `get_token()` (session lookup + lazy refresh) → `spotipy.Spotify(auth=access_token)`. No client is cached across requests; `SpotifyOAuth` is reconstructed each time it's needed (connect, callback, and inside refresh). Representative usages: `auth.me` → `sp.current_user()`; `playlist.create_playlist` → `user_playlist_create(public=True)` + chunked `user_playlist_add_tracks` (100/req); `playlist.save_discover_weekly` → paginated `current_user_playlists()` loop + `playlist_items` + add; `ai.generate` → **serial** `sp.search(type='track', limit=1)` per parsed song then create/add; `ai.save` → `current_user_follow_playlist`.

# 2. Code Quality
## Code Quality Audit — Sound Sculptor Flask Backend

Scope: technical debt, code smells, duplicated logic, dead code/imports, dead endpoints, error-handling gaps, and response-shape inconsistency. Every claim below was verified against the actual source. Security-severity items already confirmed by the lead (the tracked `.cache`, `SECRET_KEY` default, cookie token storage) are out of scope for this category except where they intersect a code-quality smell, and are not re-reported here.

### Duplicated logic — playlist creation in two places

`server/blueprints/playlist.py::create_playlist` (lines 90-119) and `server/blueprints/ai.py::generate` (lines 147-151) both implement the same "create a public playlist, build `spotify:track:` URIs, add tracks" sequence by hand. They have already drifted:

- `playlist.py` chunks the add into batches of 100 (`for i in range(0, len(uris), 100)`, lines 113-114) because Spotify caps `playlist_add_tracks` at 100 URIs per call.
- `ai.py` does a single un-chunked `sp.user_playlist_add_tracks(user_id, playlist['id'], uris)` (line 151). The AI path is documented to request "10-30 songs" so it stays under 100 today, but the divergence is a latent bug: the moment that prompt or model changes, the AI path silently fails on the 101st track while the slider path keeps working.

The URI construction `[f'spotify:track:{tid}' for tid in track_ids]` is copy-pasted verbatim (playlist.py line 110, ai.py line 150). This belongs in one helper in `server/services/spotify.py` (e.g. `create_playlist_with_tracks(sp, user_id, name, track_ids)`), used by both blueprints.

### Inconsistent response shapes — bare dict/tuple vs `jsonify`

The codebase has no convention for returning JSON. Two distinct styles coexist, sometimes inside the same file:

- Bare-dict-and-tuple returns: `auth.py` (`{'error': ...}, 400/401` and the success dict at lines 26, 33, 42, 45-49), `ai.py` (every return, lines 67/71/75/80/117/145/153/173/181), and `playlist.py::create_playlist` / `save_discover_weekly` / their error branches.
- `jsonify(...)` wrapped returns: `playlist.py::predict` (line 36) and `playlist.py::user_data` (line 87), plus `app.py::health` (line 32).

Flask 3 auto-jsonifies a returned dict, so both *work*, but the inconsistency is a maintenance smell and a correctness trap: a bare dict is auto-jsonified, but a bare **list** is NOT (Flask rejects a top-level list unless wrapped). Future code returning a list will break for whoever copies the bare-dict style. Pick one (`jsonify`) and apply it everywhere. The error envelope is at least consistent (`{'error': str}`), which is the one good thing here.

### Missing error handling on the critical AI path

`server/blueprints/ai.py::generate` makes four categories of external calls with NO try/except around any of them:

- OpenAI: `client.chat.completions.create(...)` (lines 84-103). Any `RateLimitError`, `AuthenticationError`, timeout, or network blip raises straight out → Flask 500 with a stack trace, no structured `{'error': ...}` envelope the frontend expects.
- Spotify search loop: `sp.search(...)` (line 132) runs once per recommended song with no guard.
- Spotify writes: `sp.current_user()` (148), `user_playlist_create` (149), `user_playlist_add_tracks` (151) — all unguarded.

Contrast with `playlist.py::predict` (lines 27-34), which carefully catches `FileNotFoundError`→503 and generic `Exception`→500. The AI endpoint — the headline feature, called by `AiStep.jsx` line 51 — has none of that discipline. A single Spotify token-refresh failure mid-loop produces an opaque 500. This is the worst error-handling gap in the codebase; combined with the serial-search latency already flagged by the lead, this endpoint is the most fragile in the app.

### Missing error handling on Spotify calls in auth.py and playlist.py

`get_spotify_client()` is wrapped in `try/except PermissionError` everywhere it is called (good), but the **subsequent** Spotify SDK calls are not:

- `auth.py::me` (line 44): `sp.current_user()` is unguarded. An expired-but-present token whose refresh fails, or any Spotify 5xx, raises `spotipy.SpotifyException` → unhandled 500.
- `auth.py::callback` (line 29): `oauth.get_access_token(code, check_cache=False)` is unguarded. A user who hits `/api/callback` with a stale or reused `code` (Spotify codes are single-use) gets a raw `SpotifyOauthError` → 500 instead of a clean redirect-to-error or 400.
- `playlist.py::save_discover_weekly` and `create_playlist`: every Spotify call after the `get_spotify_client()` guard (`current_user`, `current_user_playlists`, `next`, `user_playlist_create`, `playlist_items`, `user_playlist_add_tracks`) is unguarded.

Only the `PermissionError` (no-token) case is handled; the far more common "token present but Spotify call failed" case falls through to a 500.

### Dead endpoints — `/user-data` and `/save-discover-weekly`

Verified by cross-referencing `soundfrnt/src/services/api.js` against the actual call sites under `soundfrnt/src`:

- `api.js::getUserData` (line 53) → `GET /api/user-data` (`playlist.py::user_data`, line 79): defined but **never called** by any page/hook (grep for `getUserData` returns only the definition).
- `api.js::saveDiscoverWeekly` (line 56) → `POST /api/save-discover-weekly` (`playlist.py::save_discover_weekly`, line 39): defined but **never called** (grep returns only the definition).

`save_discover_weekly` (lines 39-76) is ~38 lines of non-trivial pagination/matching logic that no client invokes — pure dead weight that still has to be maintained, secured, and reasoned about. Either wire it into the UI or delete both the endpoint and its api.js method. Note also it hard-codes the English playlist name `'Discover Weekly'` (line 56), so it would silently 404 for any non-English Spotify account even if it were called.

### Unused imports

- `server/blueprints/auth.py` line 4: `get_token` is imported but never referenced in the file (only `get_spotify_oauth`, `get_spotify_client`, `TOKEN_INFO` are used). Verified via grep.
- `server/blueprints/playlist.py` line 2: `redirect` is imported but never used (only `Blueprint`, `request`, `jsonify` are used). Verified via grep.

### Dead/commented-out code blocks

- `server/services/spotify.py` lines 20-28: a fully commented-out duplicate `get_spotify_oauth` definition. It is not just noise — it documents the *intended* (default-cache) behavior that the live version (lines 8-18, with `cache_path=None, cache_handler=None`) deliberately overrode. Leaving both invites someone to "restore" the wrong one. Delete it; capture the intent in a one-line comment if needed.
- `server/config.py` lines 28-31: four commented `print(current_app.config[...])` debug lines left in the class body. Beyond being dead code, they are *broken* dead code — `current_app` (imported line 3 solely for these) has no application context at class-definition time, so uncommenting any of them raises `RuntimeError: Working outside of application context`. The `from flask import current_app` import (line 3) exists only to service these dead lines and should be removed with them.

### Trailing-whitespace artifacts in `get_spotify_oauth`

`server/services/spotify.py` lines 16-17: `cache_path=None,` and `cache_handler=None` have trailing whitespace and stray alignment. Cosmetic, but it signals the file was hand-edited under pressure (consistent with the `.cache` leak the lead found). A linter (`ruff`/`flake8`) in CI would have caught this, the unused imports, and the dead commented blocks automatically — the project ships no linter config at all.

### Inconsistent config access pattern

`ai.py` reads config via the module-level `Config` class directly: `Config.OPENAI_API_KEY` (line 73), `from server.config import Config` (line 7). Every other blueprint and service reads config through `current_app.config[...]` (the Flask-idiomatic, app-factory-respecting way — see `spotify.py` line 9, `auth.py` line 32). Reading `Config.OPENAI_API_KEY` directly bypasses any per-app override passed to `create_app(config=...)` (app.py line 7 explicitly supports an alternate config object), so a test or alternate deployment that injects a different config would be silently ignored by the AI blueprint. Use `current_app.config['OPENAI_API_KEY']` for consistency and testability.

### `_parse_songs_from_text` is brittle and silently lossy

`ai.py::_parse_songs_from_text` (lines 15-59) drives the entire AI feature off three regexes (lines 37-42) applied line-by-line. Any AI output line that doesn't match one of the three shapes is silently dropped (the `for pattern` loop just `break`s or falls through with no else). Because `generate` only errors when **zero** songs parse (line 116), a response where half the lines are in an unanticipated format produces a half-empty playlist with no warning — `total_matched`/`total_requested` (lines 158-159) are returned but nothing flags the gap. This is fragile coupling to free-text LLM output; requesting structured JSON from the model (the function already has a JSON fast-path at lines 26-32 that the prompt never asks the model to use) would make it robust. Low-severity as a quality issue but worth noting as accumulated debt on the core feature.

### Pipfile / requirements / runtime Python-version drift

`server/Pipfile` declares `python_version = "3.10"` (line 21), but the production `Dockerfile` builds on `python:3.11-slim` (line 18). Meanwhile `requirements.txt` (the file actually installed in the image, Dockerfile line 28) and `Pipfile` `[packages]` list overlapping but separately-maintained dependency sets. Three sources of truth for "what Python and which packages" that can drift independently. Pick one (requirements.txt is what Docker uses; the Pipfile appears vestigial) and delete or regenerate the other.

# 3. Reliability
## Reliability Audit — Sound Sculptor Flask Backend

This category covers crash scenarios, unhandled exceptions, race conditions, edge cases, resource exhaustion, and the nginx/gunicorn timeout mismatch. Every endpoint that touches Spotify or OpenAI was traced against the actual handler code in `server/blueprints/*` and the helpers in `server/services/*`. The headline reliability problem is systemic: **almost no Spotify or OpenAI call is wrapped in try/except**, so any upstream hiccup (rate limit, 5xx, network blip, expired token, mid-request revocation) propagates as a raw 500 with a stack trace. Several handlers can also crash on perfectly normal user accounts.

### Unhandled upstream exceptions (the dominant theme)

The helper `get_spotify_client()` (`server/services/spotify.py:50`) only ever converts the "no token in session" case into a `PermissionError`, which the blueprints catch and turn into a clean 401. Everything *after* the client is obtained — `sp.current_user()`, `sp.search()`, `sp.user_playlist_create()`, `sp.user_playlist_add_tracks()`, `sp.current_user_follow_playlist()`, `sp.next()` — is unguarded. spotipy raises `spotipy.SpotifyException` on any non-2xx (429 rate limit, 401/403 scope or revocation, 404, 5xx) and `requests.exceptions.*` on network failures. None of these are caught in `auth.py`, `playlist.py`, or `ai.py`, so they surface as bare HTTP 500s with tracebacks in logs. For an app that fans out 10–30 sequential Spotify searches per AI request (`ai.generate`), hitting a 429 at request #7 is a *likely*, not theoretical, event — and it aborts the whole request after already doing several seconds of work, with no playlist created and no useful error to the user.

The OpenAI call in `ai.generate` (`server/blueprints/ai.py:84`) is similarly naked: `client.chat.completions.create(...)` can raise `APIConnectionError`, `RateLimitError`, `APITimeoutError`, `AuthenticationError`, or `APIStatusError`. None are caught. A single OpenAI rate-limit (very common on the free/low tier this code targets with `gpt-3.5-turbo`) becomes a 500.

### Token refresh: the KeyError landmine

`get_token()` (`server/services/spotify.py:31-47`) refreshes when the token is within 60s of expiry by calling `oauth.refresh_access_token(token_info['refresh_token'])`. Two distinct failure modes: (a) `token_info['refresh_token']` is a hard subscript — if Spotify ever returned a token dict without a refresh_token (it can, on re-grants), this is a `KeyError` → 500; (b) `refresh_access_token` itself performs a network POST to Spotify and will raise `SpotifyException`/`SpotifyOauthError` if the refresh token was revoked or Spotify is down. This is the *common* "expired token mid-session" path — after the access token's ~1h lifetime, the very next request triggers an unguarded refresh, and if anything is wrong the user gets a 500 instead of a re-auth prompt.

### Crashes on normal accounts (not even error paths)

- `save_discover_weekly` (`playlist.py:71-72`) builds `uris` from `item['track']['uri']` — `item['track']` can legitimately be `None` for unavailable/local tracks; the guard `if item.get('track')` filters the outer `None` but the URI access still assumes a `uri` key. More importantly it only fetches the **first page** of `playlist_items` (no pagination), silently truncating large Discover Weekly clones — a correctness/reliability gap.
- `me`, `create_playlist`, `ai.generate`, `save_discover_weekly`, `user_data` all index `user['id']` / `sp.current_user()['id']` and `playlist['external_urls']` without defense. If `current_user()` raises (revoked token), it's a 500, not a 401, because the `PermissionError` catch only wraps the `get_spotify_client()` line, not the subsequent `sp.current_user()`.

### Empty/edge-case AI parsing

`ai.generate` does handle the two most important empty cases (`if not songs` → 502, `if not track_ids` → 404), which is good. But the regex parser (`_parse_songs_from_text`) is brittle: if OpenAI returns markdown bullets (`- "Song" by Artist`) or numbered lines without a dash, several lines won't match and silently drop. That's degraded output, not a crash, so it's lower severity — but combined with the unguarded search loop it means a partial/odd model response can still produce a tiny or empty playlist after a 500 risk.

### Concurrency / global state

The ML model uses module-global `_model`/`_y_train` with a lazy `_load_model()` and a non-atomic `if _model is not None` check (`ml.py:28-49`). Under gunicorn's default **sync** workers (`docker-entrypoint.sh` uses `--workers 2` with no `--worker-class`, no `--preload`), each worker is a single-threaded process, so within one worker there is no true thread race — but the model is loaded **per worker on first request**, duplicating the full `tracks_features.csv` read and `model.pkl` in memory across all workers, and making the *first* `/predict` after each deploy slow/janky. If anyone later switches to `gevent`/threaded workers (a one-line change), the check-then-assign becomes a genuine data race. The concurrent-token-refresh concern is real but lower-impact: two simultaneous requests from the same user can both refresh and race-write `session[TOKEN_INFO]`; last-writer-wins, and one refreshed token is discarded — not a crash, but wasted refreshes.

### Resource exhaustion / OOM

`_load_model()` does `pd.read_csv(DATA_PATH)` on the **entire** `tracks_features.csv` purely to keep the `id` column (`ml.py:48-49`). The public Spotify tracks_features dataset is ~1.2M rows / hundreds of MB; loading the whole DataFrame into pandas per worker (×2 workers) can blow the memory of a small container. And `model.pkl`/`tracks_features.csv` are not in the repo — they're bind-mounted in `docker-compose.yml:17-18`. If the host paths are absent, Docker creates **empty directories**, the mount shadows the file, `joblib.load`/`read_csv` fail, and `/predict` returns 503 forever. That's a deploy-time reliability cliff.

### nginx ↔ gunicorn timeout mismatch (confirmed)

`nginx.conf:16` sets `proxy_read_timeout 60s`; `docker-entrypoint.sh:11` runs gunicorn `--timeout 120`. The slowest path, `ai.generate`, makes 1 OpenAI call + up to 30 sequential `sp.search()` round-trips + create + add — easily exceeding 60s under load. nginx will return a **504 to the browser at 60s while gunicorn keeps working to 120s**, meaning the user sees an error but a playlist may still get created server-side (orphaned, duplicate on retry). The two timeouts must be reconciled and, ideally, the search loop parallelized/capped.

# 4. Security
## Security Audit — Sound Sculptor Flask Backend

I read every backend file (`server/app.py`, `config.py`, `run.py`, `services/spotify.py`, `services/ml.py`, all three blueprints, `requirements.txt`, `Pipfile`, `Dockerfile`, `docker-compose.yml`, `nginx.conf`, `docker-entrypoint.sh`, `.env.example`, `.gitignore`, `.dockerignore`) and the frontend `soundfrnt/src/services/api.js`. I independently re-verified the ground-truth items against the real source and git state. Everything the lead flagged held up; I found several additional concrete issues.

### Secrets management

**The committed token is the worst problem and it is real.** `git ls-files` lists `server/.cache`, and reading it shows a live Spotify `access_token` plus a long-lived `refresh_token` (and the granted `scope`). The `.gitignore` entry `.cache/` only matches a *directory* named `.cache`; the actual file is `server/.cache` (no trailing slash, in a subdirectory), so it was never ignored and got committed in `92e6245 Initial commit`. This is the spotipy `CacheFileHandler` default-write path: `get_spotify_oauth()` passes `cache_path=None, cache_handler=None`, so spotipy falls back to writing `.cache` in the process CWD. Anyone who clones the repo has the victim's refresh token and can mint new access tokens until it is revoked.

**SECRET_KEY has a hardcoded fallback.** `config.py:9` — `SECRET_KEY = os.environ.get('SECRET_KEY', 'dev-fallback-change-me')`. If `SECRET_KEY` is not set in the environment (the prod compose file does not set it — `docker-compose.yml` env only sets `FRONTEND_URL`, `CORS_ORIGINS`, `FLASK_ENV`, relying on whatever is in `.env`), Flask signs session cookies with a publicly-known key and anyone can forge a `sound_sculptor_cookie` containing arbitrary `token_info`.

**Real OpenAI/Spotify credentials would normally land in `.env`, which is correctly ignored** (`.gitignore` lines 7-9, `.dockerignore` lines 9-11), and `.env.example` ships only placeholders. So `.env` handling itself is fine — the leak is exclusively via `server/.cache`.

### Authentication & token handling

**No OAuth `state` parameter → login CSRF.** `auth.py:connect()` calls `get_spotify_oauth().get_authorize_url()` with no `state`, and `get_spotify_oauth()` (spotify.py:8-18) never sets `state` on the `SpotifyOAuth` object. I confirmed against the installed spotipy that `get_authorize_url` only emits a `state` param when one is provided — so none is sent. `callback()` (auth.py:20-33) reads only `code` and never validates `state`. An attacker can feed a victim a crafted `/api/callback?code=<attacker_code>` to bind the victim's session to the attacker's Spotify account (or the reverse), a classic OAuth login-CSRF.

**Entire token_info lives in a signed-but-unencrypted client cookie.** `callback()` does `session[TOKEN_INFO] = token_info` (auth.py:30) and `get_token()` reads/refreshes it from `session` (spotify.py:37-47). Flask's default session is `itsdangerous`-signed but **not encrypted** — the access and refresh tokens are base64-decodable by anyone who captures the cookie. Combined with the weak SECRET_KEY default, this is also forgeable.

**No explicit cookie security flags.** `config.py` sets only `SESSION_COOKIE_NAME`. There is no `SESSION_COOKIE_SECURE` (cookie will be sent over plain HTTP — and prod is HTTP on port 80 per nginx/compose), and no `SESSION_COOKIE_SAMESITE`. Flask defaults `HttpOnly=True` (so JS can't read it) and `SameSite=Lax`, but `Secure` defaults to `False`, so the token-bearing cookie travels in cleartext.

### Input validation

**`/api/create-playlist` does not validate `track_ids` element types or list size.** playlist.py:97-110 only checks that `track_ids` is non-empty, then does `f'spotify:track:{tid}'` for each element. A caller can pass non-strings or thousands of IDs; the latter becomes an unbounded number of `user_playlist_add_tracks` calls (a per-user amplification against the Spotify API on a single request).

**`/api/predict` validation is decent but unbounded numerically.** playlist.py:18-25 checks presence of all `FEATURE_KEYS` and `float()`-coerces them, which is good. There is no range clamping, but the downstream KNN won't crash on out-of-range floats, so this is minor.

**`/api/ai/generate` prompt is length-capped (500 chars, ai.py:70-71) but otherwise unvalidated** and is the expensive path (see API security).

### API security

**No rate limiting anywhere** (confirmed: no `Limiter`/`flask-limiter` in `requirements.txt`/`Pipfile`, no limiter code). `/api/ai/generate` calls OpenAI `gpt-3.5-turbo` and then runs a serial Spotify search per recommended song. An authenticated user can hammer this endpoint to run up OpenAI spend and exhaust gunicorn's 2 workers (DoS). `/api/predict` loads/holds the full CSV model in memory and is also unthrottled.

**All created playlists are forced public.** `create-playlist` (playlist.py:109) and `ai/generate` (ai.py:149) both call `user_playlist_create(..., public=True)`, and the only modify scope granted is `playlist-modify-public` (config.py:18). Users cannot get private playlists; every generated playlist is world-visible on their profile.

**Over-broad / unused scopes.** `SPOTIFY_SCOPES` requests `user-library-read`, `playlist-read-private`, `user-top-read`, `playlist-modify-public` (config.py:14-19). No endpoint uses `user-top-read` or `user-library-read` — excess privilege on the leaked token. (`playlist-read-private` is used by `save-discover-weekly`/`user-data`, which are themselves dead per the frontend.)

### CORS

`CORS(..., supports_credentials=True)` with `origins` from `CORS_ORIGINS` (app.py:14-20). This is correct as written — it does **not** use a wildcard, and prod sets `CORS_ORIGINS=http://localhost` (compose line 12). The defaults are also safe (`http://localhost:5173`). The one caveat: with `supports_credentials=True`, the origin list must never include `*`; that constraint is implicit and undocumented, so a future operator setting `CORS_ORIGINS=*` would silently break the credentialed-cookie security model. Worth a guardrail comment, not a finding on its own.

### Nginx / infra hardening

**No security response headers and no `client_max_body_size`.** `nginx.conf` sets no `X-Frame-Options`, `X-Content-Type-Options`, `Referrer-Policy`, CSP, or HSTS, and never sets `client_max_body_size`, so request body size falls back to nginx's 1 MB default — a partial accidental cap, but no intentional limit on the `/api/` proxy. Also runs plain HTTP on port 80 (no TLS), which is what makes the missing `Secure` cookie flag exploitable.

**nginx `proxy_read_timeout 60s` < gunicorn `--timeout 120s`** (nginx.conf:16 vs docker-entrypoint.sh:11). Slow AI generations return 504 at nginx while gunicorn keeps working — confirmed; reliability rather than security, included for completeness since it interacts with the unbounded AI path.

### Dev-only (not prod)

`run.py:22` uses `app.run(debug=True, ...)`, which exposes the Werkzeug interactive debugger / RCE console. This is the **dev entry only**; production runs via gunicorn (`docker-entrypoint.sh`), which never imports `run.py`. So it is a real risk only if someone runs `python -m server.run` on an exposed host — P2.

# 5. Performance
## Performance Audit — Sound Sculptor Flask Backend

Scope: production hot path is `nginx → gunicorn (2 workers) → Flask`. The user-facing endpoints actually hit by the React app (per `soundfrnt/src/services/api.js` + page call sites) are `/api/me`, `/api/ai/generate`, `/api/ai/save`, `/api/predict`, `/api/create-playlist`. `/api/user-data` and `/api/save-discover-weekly` are dead (no frontend caller), so their perf cost is latent, not active.

### 1. AI generation is dominated by serial, blocking I/O

`server/blueprints/ai.py::generate()` is the single slowest user path and it is almost entirely sequential network I/O on one gunicorn worker thread:

1. One blocking `client.chat.completions.create(model='gpt-3.5-turbo', ...)` (lines 84-103). gpt-3.5-turbo for 10-30 song lines is typically several seconds and has no `max_tokens`/timeout cap.
2. A **serial loop** doing one `sp.search(q=..., type='track', limit=1)` per parsed song (lines 125-142). The prompt asks for 10-30 songs, so this is 10-30 sequential Spotify round-trips, each ~100-400ms. That alone is commonly 2-10s wall-clock.
3. Then `sp.current_user()` (line 148), `sp.user_playlist_create()` (149), and `sp.user_playlist_add_tracks()` (151) — 3 more serial round-trips.

Total wall time for one request is the **sum** of all of these (OpenAI + ~13-33 serial Spotify calls), routinely 5-15s+. This both produces a slow UX and ties up one of only two workers for the entire duration. See finding "Serial per-song Spotify search" and the 504 finding below.

### 2. Cold-start latency on `/api/predict` from loading the whole CSV

`server/services/ml.py::_load_model()` (lines 28-49) lazy-loads on the first `/predict` after each worker boots. It does `joblib.load(MODEL_PATH)` **and** `pd.read_csv(DATA_PATH)` reading the **entire** `tracks_features.csv` into a DataFrame, then immediately discards everything except one column: `_y_train = train_data['id']` (line 49). For a Spotify-features dataset (the public "tracks_features" set is ~1.2M rows / hundreds of MB), this is a multi-second cold start and a large transient memory spike, all to keep a single string column. The first user to hit `/predict` on a fresh worker eats that latency.

### 3. Per-worker memory: model + id column resident x2 workers

After load, each worker permanently holds `_model` (the fitted KNN, which in scikit-learn stores the full training feature matrix for neighbor lookup) plus `_y_train` (the id column). With `--workers 2` (`docker-entrypoint.sh` line 11), that footprint is duplicated across both processes. Because loading is lazy and per-process, the model is also loaded twice (once per worker), and the CSV is parsed twice. There is no shared/preloaded artifact (gunicorn `--preload` is not set), so copy-on-write sharing of the model is not available either.

### 4. Worker count vs the CPU-bound prediction

`/api/predict` runs `KNN.kneighbors` (`ml.py` line 73), which is CPU-bound and, for a large training set with brute-force/exact neighbors, can be expensive. With only 2 sync workers and no thread pool, two concurrent `/predict` (or two long `/ai/generate`) requests fully occupy both workers; a third request queues behind them. The mix of one CPU-bound endpoint and several long-blocking-I/O endpoints on 2 sync workers is a poor fit — I/O-bound endpoints should use threads/async, CPU-bound should match core count.

### 5. New spotipy client + new OAuth object per request

Every authenticated request builds a brand-new `spotipy.Spotify` (`spotify.py::get_spotify_client`, line 53) and, on refresh, a new `SpotifyOAuth` (`get_spotify_oauth`, lines 8-18). spotipy clients aren't reused, so no HTTP connection pooling/keep-alive is shared across requests, and `get_spotify_oauth` calls `url_for(..., _external=True)` each time. Minor relative to the serial-search cost, but it compounds the per-call overhead in the search loop.

### 6. `save-discover-weekly` pagination cost (dead endpoint, latent)

`playlist.py::save_discover_weekly` (lines 52-62) paginates through **all** of the user's playlists with `sp.next(results)` in a loop just to find two by name — O(number of playlists / 50 per page) serial calls — and then only reads the **first page** of `playlist_items` (line 71) without paginating, so it silently truncates a >100-track Discover Weekly. It's currently unreachable from the UI, so this is latent, but it is an inefficient call pattern if revived.

### 7. No caching anywhere on dynamic responses

There is no caching layer for OpenAI results, Spotify search results, `current_user()`, or KNN predictions. `current_user()['id']` in particular is fetched fresh on every `ai.generate`, `create_playlist`, `save_discover_weekly`, and `user_data` call even though it is stable per session. Identical AI prompts re-pay the full OpenAI + search cost every time.

### Note on nginx/gunicorn timeout mismatch (perf-adjacent)

This was confirmed by the lead (`nginx.conf` `proxy_read_timeout 60s` vs gunicorn `--timeout 120`). It is directly caused by finding #1: a slow serial AI generation that runs past 60s gets 504'd by nginx while gunicorn keeps working — the user sees a failure even though the playlist may actually get created. I include it below as a P1 because it is the direct production symptom of the serial-I/O performance problem.

# 6. Maintainability
## Maintainability Audit — Sound Sculptor Flask backend

The backend is small and readable, but it is structured as a set of one-off route handlers with no shared infrastructure. The same five concerns — get-client-or-401, playlist creation, Spotify error handling, request validation, config access — are re-implemented (or omitted) in each handler. There is no test suite, no logging configuration, no schema validation, and no server-side session store. None of this crashes today, but every one of these is a place where the next change will be copy-pasted, drift, or silently break. Findings below are scoped strictly to maintainability; the P0 secret leak, cookie token storage, and timeout mismatch are owned by other categories and only referenced where they motivate a refactor.

### Separation of concerns / DRY — duplicated handler boilerplate

The `try: sp = get_spotify_client() except PermissionError: return {'error': 'Not authenticated'}, 401` block is copy-pasted **six times** across the codebase with subtly different wording (`'Not authenticated'` vs `'Not authenticated with Spotify'`): `auth.me` (auth.py:39-42), `playlist.save_discover_weekly` (playlist.py:42-45), `playlist.user_data` (playlist.py:82-85), `playlist.create_playlist` (playlist.py:103-106), `ai.generate` (ai.py:77-80), `ai.save` (ai.py:175-178). A single decorator (e.g. `@require_spotify` injecting `sp` into the handler) or a `current_user_client()` helper would collapse all six. Today a change to auth semantics (say, distinguishing expired-token from no-token) must be edited in six places.

Playlist creation is the second duplicated unit. `ai.generate` (ai.py:148-151) and `playlist.create_playlist` (playlist.py:108-114) both do `user_id = sp.current_user()['id']` → `sp.user_playlist_create(...)` → build `spotify:track:` URIs → `add_tracks`. `create_playlist` correctly chunks into batches of 100 (playlist.py:113-114) but `ai.generate` does **not** chunk (ai.py:151) — so the AI path silently breaks for >100 tracks. A shared `create_playlist_from_track_ids(sp, name, track_ids)` helper would have prevented that divergence and is the single highest-value refactor here.

### No centralized error handling — every Spotify/OpenAI call is naked

There is no Flask `@app.errorhandler` registered anywhere (app.py:7-34 has none). Each handler invents its own error contract ad hoc, and most external calls have no error handling at all:

- `auth.me` (auth.py:44) calls `sp.current_user()` with no try/except — a 401 from an expired/revoked Spotify token raises `spotipy.SpotifyException` → 500.
- `auth.callback` (auth.py:29) calls `oauth.get_access_token(...)` with no error handling — a Spotify OAuth error → 500 with a stack trace.
- `ai.generate` wraps **nothing** around the OpenAI call (ai.py:84), the per-song `sp.search` loop (ai.py:132), or `user_playlist_create`/`add_tracks` (ai.py:149-151). Any OpenAI rate-limit or Spotify hiccup → unhandled 500.
- `save_discover_weekly` (playlist.py:52, 62, 68, 71-74) and `create_playlist` (playlist.py:108-114) likewise have zero protection around Spotify calls.

Only `playlist.predict` (playlist.py:27-34) has a real try/except. The maintainability cost: error semantics are inconsistent and the only "logging" of these failures is Flask's default 500 page. A single registered `@app.errorhandler(SpotifyException)` / `@app.errorhandler(Exception)` returning a JSON envelope, plus a thin `spotify_call()` wrapper, would centralize this and give the frontend (api.js:14-17 expects `data.error`) a uniform contract.

### Logging is imported but never configured — info/exception logs are silently dropped

`ml.py:8`, `playlist.py:6`, and `ai.py:10` each create `logging.getLogger(__name__)` and emit `logger.info(...)`, `logger.error(...)`, `logger.exception(...)`. But there is **no `logging.basicConfig`, `dictConfig`, or any handler/level configuration anywhere** in the codebase (verified by grep). Under gunicorn the root logger has no application handlers, so:
- Every `logger.info` (e.g. the AI response log at ai.py:106, model-load log at ml.py:46) is **never emitted** at all (default level is WARNING).
- `logger.error`/`logger.exception` only reach output via the `logging.lastResort` fallback handler to stderr, with no timestamp, request id, or format control.

So the diagnostics the code carefully writes are mostly invisible in production. This is a real operability/maintainability gap: when a Spotify or OpenAI call fails in prod, the intended log line either doesn't fire or is unformatted. A `logging.dictConfig` (or even `logging.basicConfig`) call in `create_app` would fix it.

### Config access is inconsistent and `current_app` import is dead

Two different config-access idioms coexist: `auth.py:32` and `spotify.py:9` read `current_app.config[...]` (correct for per-app override / testability), but `ai.py:73` reads `Config.OPENAI_API_KEY` directly off the class (ai.py:7 imports `Config`). Because `Config` attributes are evaluated **at import time** from `os.environ`, the AI path can never be reconfigured per-app (e.g. in a test with a fake key) — it's pinned to whatever the environment was at import. Standardize on `current_app.config`.

Separately, `config.py:3` `from flask import current_app` is a **dead import** — it is only referenced inside the commented-out debug prints (config.py:28-31). Worse, if those prints were ever uncommented they would raise `RuntimeError: Working outside of application context` because the class body runs at import time with no app context. The dead import and the four dead comment lines should be deleted.

### No config validation on boot — misconfig fails deep inside a request

`config.py` provides silent defaults/`None` for everything: `SPOTIFY_CLIENT_ID`/`SECRET` default to `None` (config.py:12-13), `OPENAI_API_KEY` to `None` (config.py:21), `SECRET_KEY` to the literal `'dev-fallback-change-me'` (config.py:9). Nothing validates these at startup. The failure mode is deferred and obscure: a missing `CLIENT_ID` doesn't fail until `SpotifyOAuth(client_id=None, ...)` is built mid-request (spotify.py:11-12), and a missing `SECRET_KEY` silently uses the insecure fallback in prod (the lead's ground-truth #2). A `Config.validate()` / fail-fast check in `create_app` that raises on missing required prod vars would turn a confusing runtime 500 into a clear boot-time error.

### No request schema validation — hand-rolled, partial, and inconsistent

There is no pydantic/marshmallow/`flask` schema layer; every endpoint validates by hand and inconsistently:
- `predict` (playlist.py:14-25) carefully checks JSON presence, missing keys, and float-coercion.
- `create_playlist` (playlist.py:93-101) checks `track_ids` non-empty but does **no type check** — a caller sending `track_ids: "abc"` (a string) passes the truthiness check, then `[f'spotify:track:{tid}' for tid in "abc"]` (playlist.py:110) iterates characters and silently creates a playlist of garbage URIs. No validation that elements are strings/valid IDs.
- `ai.generate` (ai.py:65-71) checks prompt presence/length inline.
- `ai.save` (ai.py:169-173) reads `playlist_id` with a fallback to session.

The duplicated, ad-hoc validation is a maintainability tax and the `create_playlist` case is an actual latent bug. A declarative schema per endpoint would centralize this and eliminate the string-iteration footgun.

### Server-side session/token store — architectural debt (cross-ref security)

The entire `token_info` (access + refresh tokens) is stored in the signed-but-not-encrypted client cookie via `session[TOKEN_INFO]` (auth.py:30, refreshed in spotify.py:45). `spotify.py:16-17` deliberately disables spotipy's cache (`cache_path=None, cache_handler=None`) to force tokens into the Flask session. Beyond the security implications (owned by the security category), this is a maintainability/architecture smell: it ties token lifecycle to the cookie, caps how much can be stored, and means token refresh mutates the cookie on every request near expiry (spotify.py:42-45). A server-side session backend (Flask-Session with Redis/filesystem) or a proper token store would decouple token management from the transport and is the right long-term refactor. `ai.generate` also stashes `session['playlist_id']` (ai.py:162) as cross-request state, compounding the reliance on cookie state.

### Type safety — partial and unenforced

Type hints exist only sporadically: `ml.predict_songs(features: dict) -> list[str]` (ml.py:52) and `ai._parse_songs_from_text(text: str) -> list[dict]` (ai.py:15) are annotated, but no route handler, no `spotify.py` function (`get_token`, `get_spotify_client`, `get_spotify_oauth`), and no config is. There is no `mypy`/`pyright` config and no CI to enforce anything, so the existing hints are decorative. The `list[dict]` / `dict` hints are also un-parameterized (no key/value types). Low priority, but adopting and enforcing hints would catch the kind of shape bugs (e.g. the `track_ids` string case) the code currently ships.

### Testing — zero coverage

There are **no tests anywhere** (verified: `git ls-files` matching test/spec/conftest/pytest returns nothing; `[dev-packages]` in Pipfile is empty, requirements.txt has no pytest). For an app whose core logic is parsing (the regex song parser at ai.py:15-59 is genuinely tricky and untested), feature validation, and external-API orchestration, this is the biggest single maintainability risk: any refactor of the duplicated handlers or the parser has no safety net. The parser in particular (`_parse_songs_from_text`) is pure and trivially unit-testable, yet has no tests covering its JSON / quoted / numbered branches.

### Minor cleanup

- Dead commented-out code: the duplicate `get_spotify_oauth` (spotify.py:20-28) and the four debug `print`s in `config.py:28-31`.
- `run.py:22` runs `app.run(debug=True)` — fine for dev, but worth a guard/comment that this is dev-only (prod uses gunicorn per docker-entrypoint.sh:8). Not a prod path, so low severity.
- `requirements.txt:10` lists `gunicorn` and the Dockerfile also `pip install`s `gunicorn` explicitly (Dockerfile:29) — redundant.
- Two dependency manifests (Pipfile pins `python_version = "3.10"`, Dockerfile uses `python:3.11-slim`) — version drift between declared and actual runtime.

# 7. Functionality
## Functionality Audit — Sound Sculptor Flask Backend

I read the real source for every claim below: `server/services/ml.py`, `server/blueprints/playlist.py`, `server/blueprints/ai.py`, `server/blueprints/auth.py`, `server/services/spotify.py`, `server/config.py`, `server/app.py`, the frontend client `soundfrnt/src/services/api.js`, and the actual callers (`SliderStep.jsx`, `AiStep.jsx`, `Finished.jsx`, `useAuth.js`). The category is **Functionality** — features that are broken, incomplete, or semantically wrong.

### 1. KNN recommendation correctness — raw, unscaled features

`ml.predict_songs` (server/services/ml.py:71-74) builds the input vector directly from raw feature values and calls `_model.kneighbors` with no scaling. I confirmed the frontend feeds **real-range** values, not 0-1: `SliderStep.jsx:56-64` maps `loudness: -60 + (sliders.loudness/100)*60` (range -60..0) and `tempo: 40 + (sliders.tempo/100)*180` (range 40..220), while danceability/energy/acousticness/instrumentalness/liveness are sent as 0..1. In a Euclidean KNN, a one-step move in tempo (span 180) or loudness (span 60) swamps the entire 0..1 span of the other five dimensions. So neighbor selection is governed almost entirely by tempo and loudness; the five fine-tune sliders the UI prominently exposes are near-noise. There is no `StandardScaler`/`MinMaxScaler` in the lookup path, and nothing indicates the persisted `model.pkl` was fit on scaled data either (and if it *was* fit on scaled data, querying it with raw values is an even worse mismatch). Either way the recommendations are not what the product implies. This is the core "feature appears broken" issue for the manual path.

### 2. /predict cannot run at all in this repo state — model.pkl and tracks_features.csv are absent

`_load_model` (ml.py:35-49) raises `FileNotFoundError` if `server/model.pkl` or `server/tracks_features.csv` are missing, and `playlist.predict` (playlist.py:29-31) maps that to a `503 "ML model not available"`. I listed `server/` — neither file exists in the tree (only `.cache`, `app.py`, blueprints, services, config, etc.), and they are gitignored (`*.pkl`/`*.csv`). So out of the box the entire manual/slider flow (the headline feature, "model trained on 1M+ tracks" per `Landing.jsx:115`) returns 503. This is a deployment/packaging gap that makes a primary feature non-functional unless the artifacts are mounted at runtime — there is no fallback and nothing documents the requirement beyond the exception string.

### 3. ai/save is semantically redundant — "saving" follows a playlist the user already owns

In `ai.generate` (ai.py:148-151) the playlist is created with `sp.user_playlist_create(user_id, ..., public=True)` on the **current user's own account** at generation time — it already exists in their library/profile. Then `ai.save` (ai.py:180) calls `sp.current_user_follow_playlist(playlist_id)` to "save" it. Following a playlist you already own is a no-op in Spotify terms; the playlist was never *not* in the account. The two-step "generate then save" UX implies the playlist is provisional until saved, but it is committed to the user's account immediately. The "Save to library" button is essentially cosmetic.

### 4. The manual (ML) path also routes through ai/save, compounding the confusion

`Finished.handleSave` (Finished.jsx:72) calls `api.savePlaylist(...)` -> `POST /api/ai/save` for **both** sources. For a manual playlist the tracks were already added to a real, user-owned playlist via `createPlaylist` -> `/create-playlist` (playlist.py:90-119). So "Save to library" on the manual path is the same redundant self-follow. There is no separate save semantics; the button always does the same no-op-ish follow regardless of how the playlist was made.

### 5. save-discover-weekly is fragile and effectively dead

`save_discover_weekly` (playlist.py:39-76) finds the source by exact string match `pl['name'] == 'Discover Weekly'` (line 56). Spotify's Discover Weekly is an algorithmic, Spotify-owned playlist that does **not** reliably appear in `current_user_playlists()` unless the user has explicitly saved/followed it, and the display name is localized in non-English markets (e.g. "Descobertas da Semana", "Mix der Woche"), so the exact-name match misses it and returns `404 "Discover Weekly playlist not found"` for most users. Separately, `playlist_items` (line 71) is read **only once** with no pagination — Discover Weekly is 30 tracks so it happens to fit one page, but the code is written as if it handles arbitrary playlists and would silently truncate at 100 for any larger source. Most importantly this endpoint is **dead**: `saveDiscoverWeekly` is defined in `api.js:56` but no component imports or calls it (grep over `soundfrnt/src` shows only the definition and a comment). It is unreachable from the UI.

### 6. /user-data is a dead endpoint

`user_data` (playlist.py:79-87) returns `current_user_playlists()`. Its client wrapper `getUserData` (api.js:53) is never called anywhere in the frontend (grep confirms only the api.js definition). Dead code/endpoint.

### 7. Requested scopes don't match used features

`SPOTIFY_SCOPES` (config.py:14-19) requests `user-library-read`, `playlist-read-private`, `user-top-read`, `playlist-modify-public`. Scanning all Spotify calls across the blueprints: `current_user`, `search`, `user_playlist_create`, `user_playlist_add_tracks`, `current_user_follow_playlist`, `current_user_playlists`, `playlist_items`, `next`. 
- `user-top-read` is never used (no `current_user_top_tracks`/`_artists` anywhere) — pure over-request. 
- `user-library-read` is never used (no `current_user_saved_tracks`) — over-request. 
- `playlist-read-private` is only exercised by the dead `save-discover-weekly`/`user-data` endpoints; no live path needs it. 
- Only `playlist-modify-public` is granted (not `playlist-modify-private`), so **every** created playlist is forced `public=True` (ai.py:149, playlist.py:109) with no way to make a private one. Users get public playlists they may not want, and the app over-asks for read scopes it doesn't use, hurting the consent screen and least-privilege posture.

### 8. Robustness gaps that produce broken behavior on common inputs

- **OpenAI parse failure surfaces raw model text to the client** (ai.py:116-120): when `_parse_songs_from_text` returns nothing (gpt-3.5-turbo formatting drift is common), the endpoint returns `502` with `raw_response` echoing the entire model output. The parser only handles three quote/dash formats (ai.py:36-43); a perfectly valid markdown list like `- Song Title — Artist` without a leading number, or `**Song** by Artist`, won't match, so a successful (and paid-for) OpenAI call yields a user-facing failure. This is a frequent-path functional break, not a rare edge.
- **No try/except around any OpenAI or Spotify call in ai.generate** (ai.py:84-151): any API error (rate limit, expired token mid-loop, network) raises straight out as an unhandled 500 with no user-friendly error, and after the playlist is already created the user can be left with a partially-populated public playlist.

### 9. Leftover debug/commented code (cleanup)

- `config.py:28-31`: four commented `print(current_app.config[...])` lines sitting in the class body. They reference `current_app` at import time (would have thrown outside an app context if ever uncommented) — dead debug residue.
- `services/spotify.py:20-28`: a fully commented-out duplicate `get_spotify_oauth` definition. Dead code; the live one (lines 8-18) passes `cache_path=None, cache_handler=None` with trailing-whitespace formatting, the commented one omits those args — confusing to a future reader about which is canonical.

### Net assessment for Functionality
The two user-facing features both have correctness/semantics problems: the ML path returns tempo/loudness-dominated recommendations (and 503s with no artifacts present), and the "save" step is a redundant self-follow on both paths. Two endpoints and a whole frontend method (`save-discover-weekly`, `user-data`) are dead, the Discover Weekly clone is fragile by exact-name match, the scope set is mismatched to actual usage (all playlists forced public), and the AI path lacks error handling and has a brittle parser that fails on common LLM output.
